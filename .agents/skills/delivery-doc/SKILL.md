# Delivery Doc（交付说明文档）写作规范

参考样例：`/mnt/documents/recovery-spec-v2.md`（"错链充值找回 — 交付说明 v2"）。该样例是首选样例，新文档与它的语气、密度、章节骨架保持一致。

## 何时用

- 用户要把一批改动整理给研发 / QA / PM
- 出一个 feature 的交付说明、迭代说明、spec
- **不**用于：纯技术 RFC（更深）、营销稿（更浅）、commit log（更碎）

## 核心风格

1. **中文为主**，产品名 / 字段名 / 文件路径保持英文与 code formatting。
2. **顶部 quote block** 一句话说清文档定位：版本、覆盖范围、是否替代旧文档。例：
   > 本文档合并两批改动：v1（已上线）+ v2（最新）…研发以本文档为准；v1 中 xxx 已废弃。
3. **以"是什么"为主**，少写"为什么"；背景信息一句话带过。
4. **表格优先**——状态机、字段、对比、入口、已废弃项一律用表。
5. **三视角覆盖**：用户端流程 / 数据库 / Admin 端，少一个就少一个章节，不强行凑。
6. **绝不贴大段代码**。最多伪代码或字段示例（如 `Estimated return $Z.ZZ`）。
7. **每节短**——多数小节 5–15 行。整文档目标 100–200 行。

## 标准骨架（按需取舍，不必全用）

按编号章节组织，章节标题用名词短语：

```
# {功能名}（English Name）— 交付说明 v{N}

> 顶部定位 quote block

## 1. 功能目标         ← 必有，3–6 行说清做什么、给谁用、关键约束
## 2. 状态机 / 数据模型  ← 有状态流转就放这里，用表 + 对比 vN-1
## 3. 数据库            ← 表结构表 + RLS + Trigger 子节
## 4. 用户端流程        ← 按页面/步骤分 4.1 4.2 4.3，每个子节给入口 + 模块顺序
## 5. Admin 端          ← 若有
## 6. 已删除 / 已废弃    ← 表格列出，研发能据此清理代码
## 7. Style Guide       ← /style-guide 中的 playground 位置
## 8. 涉及文件          ← 前端 / 后端 / 数据库 分组列路径
## 9. 未变更项          ← 明确哪些没动，避免 QA 误测
```

迭代型文档（v2、v3）必有 §6 "已删除/已废弃" 和 顶部的 v1→v2 对照说明。
新 feature 首发可省略 §6 和 §9。

## 表格惯用列

| 场景 | 列 |
|---|---|
| 状态机 | status / 含义 / 触发方 / UI Badge |
| 数据库表 | 字段 / 类型 / 说明 |
| 已废弃 | 项 / 说明 |
| 对比变更 | 维度 / 旧 / 新 |

## 易错点

- 不要写"我做了 X / 我修复了 Y"——文档是给下游看的事实陈述，不是工作汇报。
- 不要堆 PR 链接、commit hash、edit-id。
- 不要在文档里描述 UI 调整过程（"先放右上后来移到左侧"），只写最终态。
- 移动端/桌面端排版差异如属本次重点，单独写一节；否则一句"移动端 / 桌面端整体布局未变"带过即可。
- 数据库章节字段列必须与 migration 一致，写之前最好 `git show` 对应 migration 文件确认。

## 输出位置

写到 `/mnt/documents/{feature-slug}-spec[-v{N}].md`，然后用 `<presentation-artifact>` 标签交付。

## 工作流

1. 用 `git log --since=... --pretty=format:"%h | %ai | %s"` 拉本次范围内的 commit
2. 对实质性 commit 用 `git show --stat` 看改动文件，跳过 "Changes" 这种空 commit
3. 按主题聚合（不要按 commit 时间线写），形成章节
4. 用骨架写第一版，自检表格密度是否够、是否有大段代码、是否覆盖三视角
5. 写入 `/mnt/documents/`，artifact 标签交付
